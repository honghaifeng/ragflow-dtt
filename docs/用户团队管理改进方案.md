# 用户管理与团队管理改进方案

## Context

现有系统中 User 和 Tenant 是 1:1 绑定（注册时 user_id == tenant_id），导致"团队"实际上就是"用户的个人空间"。权限只有 me/team 两档，角色只有 owner/normal，邀请只能按邮箱添加已注册用户。管理后台和API层各有一套独立的数据访问逻辑（management用原生SQL，API用Peewee ORM），容易不一致。本方案一次性解决这些问题，数据可重建，不需要迁移。

---

## 一、数据模型改造

### 1.1 新增 Organization 表（替代 Tenant 作为"团队"概念）

保留 Tenant 表不动（它承载 LLM 配置），新增 Organization 表作为真正的"团队/组织"概念：

```
Organization (新表)
├── id              VARCHAR(32) PK
├── name            VARCHAR(128) — 组织名称（如"电力运维部"）
├── description     TEXT — 组织描述
├── avatar          TEXT — 组织头像
├── owner_id        VARCHAR(32) FK→User — 创建者/负责人
├── create_time / create_date / update_time / update_date / status
```

### 1.2 新增 OrgMember 表（替代 UserTenant 的团队成员功能）

```
OrgMember (新表)
├── id              VARCHAR(32) PK
├── org_id          VARCHAR(32) FK→Organization
├── user_id         VARCHAR(32) FK→User
├── role            ENUM('owner', 'admin', 'editor', 'viewer') — 四级角色
├── invited_by      VARCHAR(32) FK→User
├── create_time / create_date / update_time / update_date / status
```

角色说明：
- **owner**: 组织创建者，可删除组织、管理所有成员和权限
- **admin**: 管理员，可管理成员、创建/删除知识库
- **editor**: 编辑者，可上传文档、编辑知识库内容
- **viewer**: 只读，只能查看和问答

### 1.3 新增 KbPermission 表（知识库细粒度授权）

```
KbPermission (新表)
├── id              VARCHAR(32) PK
├── kb_id           VARCHAR(32) FK→Knowledgebase
├── target_type     ENUM('user', 'org', 'public') — 授权对象类型
├── target_id       VARCHAR(32) — 用户ID或组织ID（public时为空）
├── permission      ENUM('view', 'edit', 'manage') — 权限级别
├── granted_by      VARCHAR(32) FK→User
├── create_time / create_date / update_time / update_date
```

### 1.4 修改 Knowledgebase 表

```diff
Knowledgebase:
- permission  VARCHAR(16) "me|team"
+ permission  VARCHAR(16) "private|org|custom"
  # private: 仅创建者
  # org: 所属组织全员可见
  # custom: 按 KbPermission 表精细控制
```

### 1.5 保留现有表不变

- **User 表**: 不变，注册逻辑不变
- **Tenant 表**: 保留，继续承载 LLM/模型配置（每个用户仍有一个默认 Tenant）
- **UserTenant 表**: 保留但弱化，仅用于 Tenant 的模型配置共享场景

## 二、后端 API 改造

### 2.1 新增 `api/apps/org_app.py` — 组织管理 API

路由前缀: `/api/org`

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/org/create` | POST | 创建组织（任何已登录用户） |
| `/api/org/list` | GET | 我的组织列表（我创建的+我加入的） |
| `/api/org/<org_id>` | GET | 组织详情 |
| `/api/org/<org_id>/update` | PUT | 更新组织信息（owner/admin） |
| `/api/org/<org_id>/delete` | DELETE | 删除组织（仅owner） |
| `/api/org/<org_id>/members` | GET | 成员列表 |
| `/api/org/<org_id>/member/add` | POST | 添加成员（邮箱邀请，支持批量） |
| `/api/org/<org_id>/member/<user_id>/remove` | DELETE | 移除成员 |
| `/api/org/<org_id>/member/<user_id>/role` | PUT | 修改成员角色 |
| `/api/org/<org_id>/invite-link` | POST | 生成邀请链接（带过期时间） |
| `/api/org/join/<invite_code>` | POST | 通过邀请链接加入 |

### 2.2 新增 `api/apps/kb_permission_app.py` — 知识库权限 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/kb/<kb_id>/permission/list` | GET | 查看知识库授权列表 |
| `/api/kb/<kb_id>/permission/grant` | POST | 授权（用户/组织） |
| `/api/kb/<kb_id>/permission/revoke` | DELETE | 撤销授权 |

### 2.3 修改现有权限检查逻辑

关键文件及修改点：

**`api/db/services/knowledgebase_service.py`**
- `accessible()` (line 237): 增加 KbPermission 和 OrgMember 的联查
- `accessible4deletion()` (line 264): 增加 org admin 角色检查
- `get_by_tenant_ids()` (line 73): 改造为同时返回通过组织授权可见的知识库

**`api/db/services/document_service.py`**
- `accessible()` (line 222): 增加组织权限检查
- `accessible4deletion()` (line 237): 增加 editor+ 角色检查

**`api/apps/kb_app.py`**
- 创建知识库时支持指定所属组织（org_id 字段）
- 列表接口返回权限级别标识

### 2.4 新增 ORM Service 层

在 `api/db/services/` 新增：
- `org_service.py` — OrganizationService, OrgMemberService
- `kb_permission_service.py` — KbPermissionService

### 2.5 邀请链接机制

复用现有 `InvitationCode` 表，新增字段：
```diff
InvitationCode:
+ org_id       VARCHAR(32) — 关联组织
+ expire_time  DATETIME — 过期时间
+ max_uses     INT — 最大使用次数
+ used_count   INT DEFAULT 0 — 已使用次数
```

## 三、用户端（React）改造

### 3.1 新增组织管理页面

路由: `/organization`

添加到 `web/src/routes.ts` 和导航栏 `header/index.tsx`

```
组织管理页
├── 左侧: 组织列表 + 创建组织按钮
```

## 四、管理后台（Vue3）改造

### 4.1 统一数据层：management 走 API 而非直接 SQL

**核心改造**: management server 不再直接连 MySQL 写原生 SQL，而是调用 ragflow API server 的接口（localhost:9380），管理后台只做一层 JWT 鉴权 + 转发 + 聚合。

受影响文件:
- `management/server/services/users/service.py` — 改为调用 `/api/user/*` 
- `management/server/services/teams/service.py` — 改为调用 `/api/org/*`
- `management/server/services/tenants/service.py` — 改为调用 `/api/user/tenant_info`

### 4.2 管理后台页面更新

**用户管理** `management/web/src/pages/user-management/index.vue`
- 增加"所属组织"列
- 增加"角色"列（超级管理员标识）
- 新增用户时可选择加入哪个组织

**团队管理** → 改名为**组织管理** `management/web/src/pages/team-management/index.vue`
- 重命名为 org-management
- 增加"创建组织"功能（当前只能看不能建）
- 成员管理弹窗增加角色选择（owner/admin/editor/viewer）
- 删除功能改为真正可用（当前只是 toast 提示）
- 知识库关联展示

## 六、实施顺序

1. **数据模型** — 新增 ORM 模型 + 枚举，Peewee 自动建表
2. **后端 API** — org_app.py + kb_permission_app.py + 修改权限检查
3. **用户端** — 组织管理页面 + 知识库权限改造 + 导航栏
4. **管理后台** — 统一数据层 + 页面更新
5. **集成测试** — 验证创建组织→邀请成员→创建知识库→设置权限→成员访问的完整流程

## 七、验证方式

1. 启动开发服务器 `npm start`（web 目录）
2. 注册两个测试用户 A、B
3. 用户 A 创建组织 "电力运维部"
4. 用户 A 生成邀请链接，用户 B 通过链接加入（角色 editor）
5. 用户 A 创建知识库，设置权限为 "组织可见"
6. 用户 B 登录后可看到该知识库并上传文档（editor 权限）
7. 用户 A 将 B 角色改为 viewer，B 无法再上传文档但可查看和问答
8. 管理后台登录后可查看所有组织、修改成员角色、创建新组织
